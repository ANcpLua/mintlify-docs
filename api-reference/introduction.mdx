## Authentication

The QYL API currently does not require authentication. All endpoints are accessible without API keys or bearer tokens.

<Note>
  Authentication may be added in future versions. Check the OpenAPI specification for the latest security requirements.
</Note>

## Common Response Headers

All responses include correlation headers for debugging:

| Header | Description |
|--------|-------------|
| `X-Trace-Id` | Trace ID for distributed tracing correlation |
| `X-Request-Id` | Unique request ID for support inquiries |

## Pagination

List endpoints use cursor-based pagination with the following parameters:

| Parameter | Type | Description |
|-----------|------|-------------|
| `limit` | integer | Page size (1-1000, default: 100) |
| `cursor` | string | Cursor token for next/previous page |

Responses include:

```json
{
  "items": [...],
  "has_more": true,
  "next_cursor": "abc123",
  "prev_cursor": null
}
```

## Error Responses

The API uses standard HTTP status codes:

| Code | Description |
|------|-------------|
| `200` | Success |
| `400` | Validation error - check request parameters |
| `404` | Resource not found |
| `500` | Internal server error |
| `503` | Service unavailable (health endpoints) |

Error responses follow a consistent format:

```json
{
  "code": "VALIDATION_ERROR",
  "message": "Human-readable error description",
  "details": {
    "field": "Additional context"
  }
}
```
